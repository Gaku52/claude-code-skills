# ソフトウェア開発プロセス

> 優れたソフトウェアは優れたプロセスから生まれる。ウォーターフォールからアジャイルへの変遷は、不確実性への適応力の進化である。

## この章で学ぶこと

- [ ] 主要な開発プロセスモデルを説明できる
- [ ] アジャイル開発の原則を理解する
- [ ] 現代の開発プラクティスを知る

---

## 1. 開発プロセスの進化

```
ウォーターフォール（1970年代）:
  要件定義 → 設計 → 実装 → テスト → 運用
  ↓ 各フェーズを順次完了。後戻りが困難
  問題: 要件変更に弱い。完成まで動くものが見えない

アジャイル（2001年〜）:
  ┌──────────────────────────────────────┐
  │  計画 → 設計 → 実装 → テスト → リリース │ 1-2週間
  │  計画 → 設計 → 実装 → テスト → リリース │ スプリント
  │  計画 → 設計 → 実装 → テスト → リリース │ 繰り返し
  └──────────────────────────────────────┘
  → 短いサイクルで動くソフトウェアを頻繁にリリース
  → フィードバックを素早く反映

  アジャイルマニフェスト（2001年）:
  - プロセスやツールより個人と対話
  - 包括的な文書より動くソフトウェア
  - 契約交渉より顧客との協調
  - 計画に従うより変化への対応

  Scrum:
  - スプリント（1-4週間の開発サイクル）
  - デイリースタンドアップ
  - スプリントレビュー、レトロスペクティブ
  - プロダクトバックログ、スプリントバックログ

  現代（2020年代）:
  - DevOps（開発+運用の統合）
  - CI/CD（継続的インテグレーション/デリバリー）
  - GitOps（Gitをsingle source of truthに）
  - Platform Engineering（内部開発者プラットフォーム）
```

---

## 2. 現代の開発プラクティス

```
必須の開発プラクティス:

  1. バージョン管理（Git）
     → 全てのコードをGitで管理。ブランチ戦略を統一

  2. CI/CD
     → プッシュごとに自動テスト、自動デプロイ
     → GitHub Actions, GitLab CI, CircleCI

  3. コードレビュー
     → プルリクエストで変更をレビュー
     → 品質向上 + 知識共有

  4. テスト自動化
     → ユニットテスト、統合テスト、E2Eテスト
     → テストカバレッジの継続的な計測

  5. Infrastructure as Code
     → Terraform, CloudFormation
     → インフラを宣言的に管理

  6. モニタリング・オブザーバビリティ
     → メトリクス、ログ、トレース
     → Prometheus, Grafana, Datadog
```

## まとめ

| 概念 | ポイント |
|------|---------|
| ウォーターフォール | 順次実行。変更に弱い |
| アジャイル | 反復・漸進的。変化に適応 |
| Scrum | 1-4週スプリント。役割とイベントが定義 |
| DevOps | 開発と運用の統合。CI/CDが基盤 |
| 現代 | GitOps, Platform Engineering |
