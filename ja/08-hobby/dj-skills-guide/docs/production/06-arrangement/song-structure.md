# Song Structure（曲構造）

8つのセクションで完璧な楽曲を設計します。Intro・Verse・Buildup・Drop・Breakdownの配置を完全マスターします。

## この章で学ぶこと

- 8つのセクション詳細
- セクション別役割
- 長さの決め方
- Techno/House/Hip Hop構成
- タイムライン設計
- セクション配置戦略
- よくあるパターン


## 前提知識

このガイドを読む前に、以下の知識があると理解が深まります:

- 基本的なプログラミングの知識
- 関連する基礎概念の理解
- [Intro & Outro（イントロとアウトロ）](./intro-outro.md) の内容を理解していること

## 8つのセクション

**基本構成:**

### 1. Intro（イントロ）

```
目的:

楽曲への導入
DJ対応
雰囲気作り

長さ:

Techno/House: 32小節 (必須)
Hip Hop: 8小節
Pop: 4-8小節

理由:
DJミックス 16-32小節必要

エネルギー:

2-3/10
低い
徐々に上昇

要素:

最小限:
Bar 1-8: Kick・Hi-Hat
Bar 9-16: Bass追加
Bar 17-24: Percussion追加
Bar 25-32: Pad追加

メロディー:
なし (Intro)

Vocal:
なし

重要ルール:

Intro = Outro
同じ要素推奨

理由:
DJがループ可能
ミックスしやすい

Techno Intro例:

Bar 1-8:
Kick (4つ打ち)
Hi-Hat (16分音符)

Bar 9-16:
Bass追加
シンプルパターン

Bar 17-24:
Shaker追加
Percussion追加

Bar 25-32:
Pad追加
雰囲気作り
→ Verse へ
```

### 2. Verse（ヴァース）

```
目的:

メインセクション前
ストーリー展開
Vocal (ある場合)

長さ:

Techno/House: 16-32小節
Hip Hop: 16小節 (典型的)
Pop: 8-16小節

エネルギー:

4-5/10
中程度
準備段階

要素:

ほぼ全て:
Kick・Bass・Drums
Vocal・Lead (控えめ)
Pad・FX

まだ控えめ:
Leadフル開放なし
Filter少し暗め

Hip Hop Verse:

特徴:
ラップ
16小節 (8+8)
シンプルなBeat

Techno Verse:

特徴:
メロディー控えめ
グルーヴ重視
32小節

役割:

Buildup への準備
聴き手を温める
エネルギー徐々に上昇
```

### 3. Buildup（ビルドアップ）

```
目的:

緊張感
期待感
Drop への準備

長さ:

8小節 (最も一般的)
16小節 (長め)
4小節 (短め、まれ)

推奨: 8小節

エネルギー:

5 → 9/10
急上昇
最大の期待感

要素:

Filter上昇:
Cutoff 300 Hz → 8000 Hz
Resonance 20% → 70%

White Noise:
Riser
徐々に大きく

Snare Roll:
Bar 7-8
16分音符・32分音符

Kick:
Bar 8.4: 消える (瞬間停止)
または維持

Automation:

Send上昇:
Reverb 25% → 60%

Volume上昇:
FX徐々に大きく

8小節Buildup構成:

Bar 1-4 (Phase 1):
Filter 300 → 1000 Hz
White Noise出現

Bar 5-6 (Phase 2):
Filter 1000 → 5000 Hz
Snare細かく

Bar 7-8 (Phase 3):
Filter 5000 → 8000 Hz
Snare Roll全開
Bar 8.4: Kick消える

→ Drop!

重要:

最後の瞬間:
全て停止 (0.25拍)
または
Lowpass Filter 0 Hz

効果:
最大のインパクト
```

### 4. Drop（ドロップ）

```
目的:

最大の盛り上がり
解放感
楽曲のハイライト

長さ:

16-32小節
通常: 16小節 (1st Drop)
24-32小節 (2nd Drop)

エネルギー:

10/10
最大

要素:

全て:
Kick (復帰・全開)
Bass (最大音量)
Lead (全開)
Drums (全て)
Vocal (ある場合)
FX

Filter:

Bypass
または
Cutoff全開 (8000 Hz+)

1st Drop vs 2nd Drop:

1st Drop (Bar 73-88):
16小節
標準

2nd Drop (Bar 129-160):
32小節
より長い
FX追加
より豪華

Drop構成例:

Bar 1-8:
全要素
最大音量

Bar 9-16:
継続
または
要素少し変化

Bar 17-24 (2nd Dropのみ):
FX追加
Vocal追加

Bar 25-32 (2nd Dropのみ):
徐々に Outro へ移行

重要:

Drop = 楽曲の顔
最も記憶に残る
最高の音質必須
```

### 5. Breakdown（ブレイクダウン）

```
目的:

休息
対比
感動・空気感

長さ:

16-32小節
通常: 16小節

エネルギー:

3-4/10
低い
休息

要素:

最小限:
Vocal (ある場合)
Pad
FX・Ambience

削除:
Kick (完全に消える)
Bass (消える)
Drums (ほぼ消える)

効果:

対比:
Drop → Breakdown
10/10 → 3/10
劇的変化

感情:
静寂
感動的
空間

次への準備:
Buildup 2 へ
再びエネルギー上昇

Breakdown構成例:

Bar 1-8:
Kick・Bass消える
Vocal・Padのみ
静寂

Bar 9-16:
徐々に要素追加
Hi-Hat復帰
Percussion復帰
→ Buildup 2 へ

ジャンル別:

Techno:
32小節Breakdown
ゆっくり

House:
16小節Breakdown
Vocal中心

Hip Hop:
Breakdownなし
または Bridge
```

### 6. Bridge（ブリッジ）

```
目的:

変化
新しいセクション
Chorus への移行

使用:

Hip Hop・Pop: よく使う
Techno・House: まれ

長さ:

8小節
16小節

エネルギー:

5-6/10
中程度

要素:

異なるコード進行
異なるメロディー
変化

Hip Hop Bridge例:

Bar 1-8:
コード変化
メロディー変化
Vocal変化
→ Chorus へ

Techno:

Breakdownが代わり
Bridge まれ
```

### 7. Chorus（コーラス）

```
目的:

サビ
最も記憶に残る
繰り返し

使用:

Pop・Hip Hop: 必須
House: よく使う (Vocal)
Techno: まれ (Drop が代わり)

長さ:

8小節
16小節

エネルギー:

8-10/10
高い

要素:

全て
Vocal (キャッチー)
メインメロディー

繰り返し:

通常 2-3回
曲中で繰り返す

例:

Intro → Verse → Chorus
→ Verse → Chorus
→ Bridge → Chorus
→ Outro
```

### 8. Outro（アウトロ）

```
目的:

楽曲の終了
DJ対応
次の曲へ

長さ:

Techno/House: 32小節 (必須)
Hip Hop: 8小節
Pop: 4-8小節

エネルギー:

10 → 2/10
徐々に下降

要素:

徐々に削除:
Bar 1-8: Lead・Vocal消える
Bar 9-16: Pad・FX消える
Bar 17-24: Percussion消える
Bar 25-32: Kick・Hi-Hat・Bass のみ

最後:
Kick・Hi-Hat・Bass
次の曲とミックス可能

Intro との関係:

Outro = Intro (逆)
同じ要素

理由:
DJがループ可能
Outro → 別曲 Intro
シームレス

Techno Outro例:

Bar 145-152:
Lead消える
Vocal消える

Bar 153-160:
Pad消える
FX消える

Bar 161-168:
Percussion消える

Bar 169-176:
Kick・Hi-Hat・Bass のみ
ループ可能
→ 次の曲へ
```

## セクション配置戦略

**原則:**

### 1. 対比の法則

```
高 → 低:
Drop → Breakdown
劇的

低 → 高:
Breakdown → Buildup → Drop
感動的

繰り返し:
同じエネルギー続かない

推奨パターン:

低 (Intro)
→ 中 (Verse)
→ 高 (Drop)
→ 低 (Breakdown)
→ 高 (Drop 2)
→ 低 (Outro)

波型:
理想的
```

### 2. 8小節ルール

```
全セクション:
8の倍数

理由:
DJミックス
人間の認知
業界標準

例外:

Pop:
4小節あり

Hip Hop:
16小節Verse標準

Techno/House:
8小節厳守
```

### 3. 2回ピークルール

```
1回のみ:
物足りない

2回:
理想的
Drop 1・Drop 2

3回:
可能だが長くなる
7-8分

推奨:
2回のDrop
```

from typing import List, Dict, Optional
from datetime import datetime

class AdvancedExercise:
    """応用パターンの演習"""

    def __init__(self, max_size: int = 100):
        self._items: List[Dict] = []
        self._max_size = max_size
        self._created_at = datetime.now()

    def add(self, key: str, value: any) -> bool:
        """アイテムの追加（サイズ制限付き）"""
        if len(self._items) >= self._max_size:
            return False
        self._items.append({
            'key': key,
            'value': value,
            'timestamp': datetime.now().isoformat()
        })
        return True

    def find(self, key: str) -> Optional[Dict]:
        """キーによる検索"""
        for item in reversed(self._items):
            if item['key'] == key:
                return item
        return None

    def remove(self, key: str) -> bool:
        """キーによる削除"""
        for i, item in enumerate(self._items):
            if item['key'] == key:
                self._items.pop(i)
                return True
        return False

    def stats(self) -> Dict:
        """統計情報"""
        return {
            'total_items': len(self._items),
            'max_size': self._max_size,
            'usage_percent': len(self._items) / self._max_size * 100,
            'uptime': str(datetime.now() - self._created_at)
        }

# テスト
def test_advanced():

    assert ex.add("a", 1) == True
    assert ex.add("b", 2) == True
    assert ex.add("c", 3) == True
    assert ex.add("d", 4) == False  # サイズ制限
    assert ex.find("b")['value'] == 2
    assert ex.remove("b") == True
    assert ex.find("b") is None

# 最適化前（O(n^2)）
def slow_search(data: list, target: int) -> int:
    """非効率な検索"""
    for i in range(len(data)):
        for j in range(i + 1, len(data)):
            if data[i] + data[j] == target:
                return (i, j)
    return (-1, -1)

# 最適化後（O(n)）
def fast_search(data: list, target: int) -> tuple:
    """ハッシュマップを使った効率的な検索"""
    seen = {}
    for i, num in enumerate(data):
        complement = target - num
        if complement in seen:
            return (seen[complement], i)
        seen[num] = i
    return (-1, -1)

### デバッグの手順

1. **エラーメッセージの確認**: スタックトレースを読み、発生箇所を特定する
2. **再現手順の確立**: 最小限のコードでエラーを再現する
3. **仮説の立案**: 考えられる原因をリストアップする
4. **段階的な検証**: ログ出力やデバッガを使って仮説を検証する
5. **修正と回帰テスト**: 修正後、関連する箇所のテストも実行する

    level=logging.DEBUG,

def debug_decorator(func):
    """関数の入出力をログ出力するデコレータ"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        logger.debug(f"呼び出し: {func.__name__}(args={args}, kwargs={kwargs})")
        try:
            result = func(*args, **kwargs)
            logger.debug(f"戻り値: {func.__name__} -> {result}")
            return result
        except Exception as e:
            logger.error(f"例外発生: {func.__name__}: {e}")
            logger.error(traceback.format_exc())
            raise
    return wrapper

@debug_decorator
def process_data(items):

### トレードオフの分析

技術的な判断には必ずトレードオフが伴います。以下の観点で分析を行いましょう:

**1. 短期 vs 長期のコスト**
- 短期的に速い方法が長期的には技術的負債になることがある
- 逆に、過剰な設計は短期的なコストが高く、プロジェクトの遅延を招く

**2. 一貫性 vs 柔軟性**
- 統一された技術スタックは学習コストが低い
- 多様な技術の採用は適材適所が可能だが、運用コストが増加

**3. 抽象化のレベル**
- 高い抽象化は再利用性が高いが、デバッグが困難になる場合がある
- 低い抽象化は直感的だが、コードの重複が発生しやすい

```python
# 設計判断の記録テンプレート
class ArchitectureDecisionRecord:
    """ADR (Architecture Decision Record) の作成"""

    def __init__(self, title: str):
        self.title = title
        self.context = ""
        self.decision = ""
        self.consequences = []
        self.alternatives = []

    def set_context(self, context: str):
        """背景と課題の記述"""
        self.context = context
        return self

    def set_decision(self, decision: str):
        """決定内容の記述"""
        self.decision = decision
        return self

    def add_consequence(self, consequence: str, positive: bool = True):
        """結果の追加"""
        self.consequences.append({
            'description': consequence,
            'type': 'positive' if positive else 'negative'
        })
        return self

    def add_alternative(self, name: str, reason_rejected: str):
        """却下した代替案の追加"""
        self.alternatives.append({
            'name': name,
            'reason_rejected': reason_rejected
        })
        return self

    def to_markdown(self) -> str:
        """Markdown形式で出力"""
        md = f"# ADR: {self.title}\n\n"
        md += f"## 背景\n{self.context}\n\n"
        md += f"## 決定\n{self.decision}\n\n"
        md += "## 結果\n"
        for c in self.consequences:
            icon = "✅" if c['type'] == 'positive' else "⚠️"
            md += f"- {icon} {c['description']}\n"
        md += "\n## 却下した代替案\n"
        for a in self.alternatives:
            md += f"- **{a['name']}**: {a['reason_rejected']}\n"
        return md
```

## よくある構成ミスと修正法

**初心者の落とし穴:**

### ミス1: Intro/Outro短すぎ

```
問題:
Intro 8小節
Outro 8小節

結果:
DJミックス不可能
プレイされない

修正:

Intro: 8小節 → 32小節に延長
Outro: 8小節 → 32小節に延長

方法:

Bar 1-8: Kick・Hi-Hat
Bar 9-16: Kick・Hi-Hat (同じ)
Bar 17-24: Bass追加
Bar 25-32: Percussion追加

結論:
32小節必須 (Techno/House)
```

### ミス2: Buildup長すぎ

```
問題:
Buildup 32小節

結果:
聴き手飽きる
緊張感ピーク過ぎる
Drop弱い

修正:

32小節 → 8小節に短縮

または

32小節を分割:
Bar 1-16: Verse (ビルド的)
Bar 17-24: Buildup (真のBuildup・8小節)

結論:
Buildup 8小節が理想
最大16小節まで
```

### ミス3: Dropバリエーションなし

```
問題:
Drop 1 = Drop 2
完全に同じ

結果:
2回目物足りない
予測可能
飽きる

修正:

Drop 2に追加:
FX (Riser・Impact)
Vocal (ある場合)
異なるLead旋律
ドラムフィル

または

Drop 2延長:
16小節 → 32小節

結論:
Drop 2 = Drop 1 + α
より豪華に
```

### ミス4: 8小節ルール無視

```
問題:
Buildup 7小節
Verse 13小節
Breakdown 20小節

結果:
違和感
DJミックス困難
リズム崩れる

修正:

7小節 → 8小節
13小節 → 16小節
20小節 → 16 or 24小節

結論:
必ず8の倍数
例外なし (Techno/House)
```
