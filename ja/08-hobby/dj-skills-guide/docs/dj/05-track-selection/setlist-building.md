## 1. 60分セットの基本構造

### 4つのフェーズ

**Phase 1: ウォームアップ (0-15分)**
```
目的: 観客を温める

特徴:
- 低〜中エネルギー
- 遅めのBPM
- メロウな曲
- グルーヴ重視

曲数: 3-5曲
```

**Phase 2: エネルギー構築 (15-35分)**
```
目的: 徐々にエネルギーを上げる

特徴:
- 中〜高エネルギー
- BPMを徐々に上げる
- ビルドアップ
- ダンサブル

曲数: 5-7曲
```

**Phase 3: ピーク (35-50分)**
```
目的: 最高潮

特徴:
- 最高エネルギー
- 速いBPM
- ドロップ多用
- 盛り上がり曲

曲数: 4-6曲
```

**Phase 4: クールダウン (50-60分)**
```
目的: 次のDJへの橋渡し

特徴:
- エネルギー下げる
- BPM下げる
- スムーズな終わり方

曲数: 2-4曲
```

## 2. BPMの流れ

### House セット例

**時間別BPM:**
```
0-15分:   120-123 BPM (ウォームアップ)
15-25分:  123-126 BPM (徐々に上げる)
25-40分:  126-128 BPM (ピーク前)
40-50分:  128-130 BPM (ピーク)
50-60分:  126-128 BPM (クールダウン)

ルール: +2-3 BPMずつ上げる
```

### Techno セット例

**時間別BPM:**
```
0-15分:   125-128 BPM
15-35分:  128-132 BPM
35-50分:  132-138 BPM (ピーク)
50-60分:  130-132 BPM

Techno: Houseより速め
```

### Hip Hop セット例

**時間別BPM:**
```
全体: 85-100 BPM (変化が少ない)

理由:
- Hip HopはBPM変化が小さい
- エネルギーは選曲とスクラッチで調整
- グルーヴ重視
```

## 3. キーの流れ

### Camelot Wheel復習

**ハーモニックミキシング:**
```
基本ルール:
- 同じキー: 8A + 8A
- ±1: 8A + 7A または 9A
- A ↔ B: 8A + 8B

60分セットのキーの流れ例:
8A → 8A → 9A → 9B → 10B → 10A → 9A → 8A

円を描くように移動
```

### キーの流れ例

**House セット:**
```
0-15分:   8A (Am) - Deep House
15-25分:  8A → 9A (Em) - 徐々に上げる
25-35分:  9A → 10A (Bm) - エネルギー上げる
35-45分:  10A → 10B (D) - ピーク
45-50分:  10B → 9B (A) - まだピーク
50-60分:  9A → 8A - クールダウン

マイナーキーから始め、ピークでメジャーキー
```

## 5. 時間配分の計算

### 平均ミックス時間

**ジャンル別:**
```
House/Techno:
- ウォームアップ: 5-7分/曲
- ピーク: 3-4分/曲
- クールダウン: 5-7分/曲

Hip Hop:
- 2-3分/曲（短め）
- スクラッチで延長

計算: 60分 ÷ 平均時間 = 曲数
```

### 時間調整の方法

**長すぎる場合:**
```
1. ミックス時間を短くする（8 bar → 16 bar）
2. イントロを短く
3. アウトロを短く
4. 曲数を減らす

目標: 58-62分（±2分許容範囲）
```

**短すぎる場合:**
```
1. ミックス時間を長くする（16 bar → 32 bar）
2. ループで延長
3. 曲数を増やす
```

## 7. よくある失敗と対処法

### 失敗1: BPMの上げすぎ

**問題:**
```
120 → 130 → 140 BPM（急上昇）
→ 観客がついていけない
```

**対処:**
```
✓ +2-3 BPMずつ
✓ 5曲で+10 BPM程度
✓ 段階的に
```

### 失敗2: キーがバラバラ

**問題:**
```
8A → 3B → 11A → 5B
→ 不協和音、違和感
```

**対処:**
```
✓ Camelot Wheelを守る
✓ ±1キーずつ移動
✓ 事前にチェック
```

### 失敗3: ピークが多すぎる

**問題:**
```
全ての曲がピーク曲
→ メリハリがない
→ 観客が疲れる
```

**対処:**
```
✓ ピークは1-2回
✓ 波を作る
✓ クールダウンも重要
```

### 失敗4: 予備曲がない

**問題:**
```
観客の反応が悪い
→ 変更したい
→ 予備がない
→ パニック
```

**対処:**
```
✓ 各フェーズに予備2-3曲
✓ 合計10曲の予備
✓ 柔軟に対応
```

### 演習2: 応用パターン

基本実装を拡張して、以下の機能を追加してください。

```python
# 演習2: 応用パターン
from typing import List, Dict, Optional
from datetime import datetime

class AdvancedExercise:
    """応用パターンの演習"""

    def __init__(self, max_size: int = 100):
        self._items: List[Dict] = []
        self._max_size = max_size
        self._created_at = datetime.now()

    def add(self, key: str, value: any) -> bool:
        """アイテムの追加（サイズ制限付き）"""
        if len(self._items) >= self._max_size:
            return False
        self._items.append({
            'key': key,
            'value': value,
            'timestamp': datetime.now().isoformat()
        })
        return True

    def find(self, key: str) -> Optional[Dict]:
        """キーによる検索"""
        for item in reversed(self._items):
            if item['key'] == key:
                return item
        return None

    def remove(self, key: str) -> bool:
        """キーによる削除"""
        for i, item in enumerate(self._items):
            if item['key'] == key:
                self._items.pop(i)
                return True
        return False

    def stats(self) -> Dict:
        """統計情報"""
        return {
            'total_items': len(self._items),
            'max_size': self._max_size,
            'usage_percent': len(self._items) / self._max_size * 100,
            'uptime': str(datetime.now() - self._created_at)
        }

# テスト
def test_advanced():
    ex = AdvancedExercise(max_size=3)
    assert ex.add("a", 1) == True
    assert ex.add("b", 2) == True
    assert ex.add("c", 3) == True
    assert ex.add("d", 4) == False  # サイズ制限
    assert ex.find("b")['value'] == 2
    assert ex.remove("b") == True
    assert ex.find("b") is None
    stats = ex.stats()
    assert stats['total_items'] == 2
    print("応用テスト全合格!")

test_advanced()
```

```python
# 演習3: パフォーマンス最適化
import time
from functools import lru_cache

# 最適化前（O(n^2)）
def slow_search(data: list, target: int) -> int:
    """非効率な検索"""
    for i in range(len(data)):
        for j in range(i + 1, len(data)):
            if data[i] + data[j] == target:
                return (i, j)
    return (-1, -1)

# 最適化後（O(n)）
def fast_search(data: list, target: int) -> tuple:
    """ハッシュマップを使った効率的な検索"""
    seen = {}
    for i, num in enumerate(data):
        complement = target - num
        if complement in seen:
            return (seen[complement], i)
        seen[num] = i
    return (-1, -1)

# ベンチマーク

    data = list(range(5000))
    random.shuffle(data)
    target = data[100] + data[4000]

    result1 = slow_search(data, target)
    slow_time = time.time() - start

    start = time.time()
    result2 = fast_search(data, target)
    fast_time = time.time() - start

    print(f"非効率版: {slow_time:.4f}秒")
    print(f"効率版:   {fast_time:.6f}秒")
```

## トラブルシューティング

### よくあるエラーと解決策

| エラー | 原因 | 解決策 |
|--------|------|--------|
| 初期化エラー | 設定ファイルの不備 | 設定ファイルのパスと形式を確認 |
| タイムアウト | ネットワーク遅延/リソース不足 | タイムアウト値の調整、リトライ処理の追加 |
| メモリ不足 | データ量の増大 | バッチ処理の導入、ページネーションの実装 |
| 権限エラー | アクセス権限の不足 | 実行ユーザーの権限確認、設定の見直し |
| データ不整合 | 並行処理の競合 | ロック機構の導入、トランザクション管理 |

### デバッグの手順

1. **エラーメッセージの確認**: スタックトレースを読み、発生箇所を特定する
2. **再現手順の確立**: 最小限のコードでエラーを再現する
3. **仮説の立案**: 考えられる原因をリストアップする
4. **段階的な検証**: ログ出力やデバッガを使って仮説を検証する
5. **修正と回帰テスト**: 修正後、関連する箇所のテストも実行する

```python
# デバッグ用ユーティリティ
import logging
import traceback
from functools import wraps

# ロガーの設定
logging.basicConfig(
    level=logging.DEBUG,
    format='%(asctime)s [%(levelname)s] %(name)s: %(message)s'
)
logger = logging.getLogger(__name__)

def debug_decorator(func):
    """関数の入出力をログ出力するデコレータ"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        logger.debug(f"呼び出し: {func.__name__}(args={args}, kwargs={kwargs})")
        try:
            result = func(*args, **kwargs)
            logger.debug(f"戻り値: {func.__name__} -> {result}")
            return result
        except Exception as e:
            logger.error(f"例外発生: {func.__name__}: {e}")
            logger.error(traceback.format_exc())
            raise
    return wrapper

@debug_decorator
def process_data(items):
    """データ処理（デバッグ対象）"""
    if not items:
        raise ValueError("空のデータ")
    return [item * 2 for item in items]
```

### トレードオフの分析

技術的な判断には必ずトレードオフが伴います。以下の観点で分析を行いましょう:

**1. 短期 vs 長期のコスト**
- 短期的に速い方法が長期的には技術的負債になることがある
- 逆に、過剰な設計は短期的なコストが高く、プロジェクトの遅延を招く

**2. 一貫性 vs 柔軟性**
- 統一された技術スタックは学習コストが低い
- 多様な技術の採用は適材適所が可能だが、運用コストが増加

**3. 抽象化のレベル**
- 高い抽象化は再利用性が高いが、デバッグが困難になる場合がある
- 低い抽象化は直感的だが、コードの重複が発生しやすい

```python
# 設計判断の記録テンプレート
class ArchitectureDecisionRecord:
    """ADR (Architecture Decision Record) の作成"""

    def __init__(self, title: str):
        self.title = title
        self.context = ""
        self.decision = ""
        self.consequences = []
        self.alternatives = []

    def set_context(self, context: str):
        """背景と課題の記述"""
        self.context = context
        return self

    def set_decision(self, decision: str):
        """決定内容の記述"""
        self.decision = decision
        return self

    def add_consequence(self, consequence: str, positive: bool = True):
        """結果の追加"""
        self.consequences.append({
            'description': consequence,
            'type': 'positive' if positive else 'negative'
        })
        return self

    def add_alternative(self, name: str, reason_rejected: str):
        """却下した代替案の追加"""
        self.alternatives.append({
            'name': name,
            'reason_rejected': reason_rejected
        })
        return self

    def to_markdown(self) -> str:
        """Markdown形式で出力"""
        md = f"# ADR: {self.title}\n\n"
        md += f"## 背景\n{self.context}\n\n"
        md += f"## 決定\n{self.decision}\n\n"
        md += "## 結果\n"
        for c in self.consequences:
            icon = "✅" if c['type'] == 'positive' else "⚠️"
            md += f"- {icon} {c['description']}\n"
        md += "\n## 却下した代替案\n"
        for a in self.alternatives:
            md += f"- **{a['name']}**: {a['reason_rejected']}\n"
        return md
```

## 10. セットリスト構築の高度な理論

ここからは基本を超えた、プロフェッショナルレベルのセットリスト構築技術を解説します。60分セットの基本を理解した上で、より長いセット、より複雑な構成、そしてさまざまなシチュエーションに対応するための高度なテクニックを学びましょう。

### 10.1 ナラティブ・アーク理論（物語としてのセット）

セットリストを「物語」として捉えることで、観客を感情的な旅に連れ出すことができます。映画の脚本と同様に、DJセットにも起承転結があります。

```
│     世界観の提示        ジャンル・ムードの提示              │
│                                                         │
│  2. 事件（展開）   →  エネルギー構築                      │
│     緊張感の上昇        BPM上昇、テンション増加            │
```

### 10.3 フロアリーディングとセットリスト修正

事前に準備したセットリストを、リアルタイムのフロアの状況に応じて修正する技術です。

**フロアリーディングの指標:**
```
観察ポイント:
┌──────────────────────────────────────────────────┐
│ 指標              良好            要修正          │
├──────────────────────────────────────────────────┤
│ ダンス人数       増加傾向        減少傾向         │
│ 体の動き         大きい          小さい/停止      │
│ 表情             笑顔/恍惚       無表情/退屈      │
│ バーカウンター   空いている      混んでいる        │
│ スマホ率         低い            高い              │
│ 手を上げる人     多い            少ない            │
│ 声援・歓声       あり            なし              │
│ フロア位置       前方に集中      後方に分散        │
└──────────────────────────────────────────────────┘
```

**セットリスト修正のフローチャート:**
```
フロアの反応を観察
        │
        ▼
┌─ 反応良好 ──→ セットリスト通り続行
│
├─ 反応普通 ──→ 次の曲で微調整
│                  ├─ BPMを少し上げる
│                  ├─ よりキャッチーな曲に差し替え

│                  ├─ BPMを大幅に変更
│                  └─ 定番曲・アンセムを投入
│
└─ フロアが空 ──→ 緊急対応
                   ├─ エネルギーを大幅に下げる
                   ├─ 聴き馴染みのある曲を選ぶ
                   └─ ウォームアップからやり直す
```

## 12. エネルギーカーブの詳細設計

### 12.1 エネルギーレベルの数値化

```
- 使用場面: 序盤、夕方のテラス
```

### 13.3 3時間セットの設計

3時間セットはヘッドライナーやレジデントDJに求められる究極のフォーマットです。マラソンのような持久力と、映画のような構成力が必要です。

**3時間セット構造概要:**
```
全体構成（映画的アプローチ）:

時間        役割              エネルギー  曲数
0:00-45     プロローグ        1-4/10     10-15
45-90       第1幕（展開）     4-7/10     12-18
90-120      インターミッション 5-6/10     6-10
120-150     第2幕（上昇）     6-8/10     10-14
150-165     クライマックス    8-10/10    6-10
165-180     エピローグ        5-2/10     5-8

合計: 約50-75曲 + 予備30曲 = 約80-105曲の準備が必要

3時間セット特有の注意点:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 体力管理: 立ちっぱなし3時間は体力的にきつい
   → 水分補給、姿勢の変化を意識
2. 集中力維持: 後半にミスが出やすい
   → 後半は準備度の高い曲を配置
3. 曲の在庫: 最低100曲の持ち曲が必要
   → 日頃からのディグが重要
4. 観客の入れ替わり: 3時間で観客層が変化する
   → 時間帯に応じたジャンル調整
5. 次のDJとの調整: 前後のDJとの打ち合わせ必須
   → BPM/ジャンルの引き継ぎ確認
```

## 14. B2Bセットリストの組み方

### 14.1 B2Bセットとは

B2B（Back to Back）セットは、2人以上のDJが交互にプレイするスタイルです。化学反応が生まれる一方で、コミュニケーション不足だと崩壊するリスクもあります。

**B2Bの基本フォーマット:**
```
交代パターン:

パターン1: 1曲交代（最も一般的）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DJ-A: 曲1 → DJ-B: 曲2 → DJ-A: 曲3 → DJ-B: 曲4 ...
  ↓      ↑      ↓      ↑
ミックスアウト  ミックスアウト

メリット: テンポが良い、予測不能な展開
デメリット: 準備時間が短い、プレッシャー大

パターン2: 2曲交代
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DJ-A: 曲1,2 → DJ-B: 曲3,4 → DJ-A: 曲5,6 → ...

メリット: 余裕がある、自分の流れを作れる
デメリット: 交代の波が大きくなりがち

パターン3: 時間交代（15-30分ずつ）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DJ-A: 0-30分 → DJ-B: 30-60分 → DJ-A: 60-90分 ...

メリット: 各自のスタイルを出しやすい
デメリット: B2B感が薄い、ほぼリレー

パターン4: ハイブリッド
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ウォームアップ: 2曲交代
ピーク: 1曲交代
クールダウン: 2曲交代
```

### 14.2 B2B成功の鍵

**事前準備チェックリスト:**
```
□ 共通のジャンル/BPM範囲を決める
□ 共有プレイリストを作成（各自30-50曲提出）
□ 「絶対に使いたい曲」を5曲ずつ共有
□ 交代パターンを決定
□ 合図（アイコンタクト、手の動き）を決める
□ 機材セッティングを確認（USB2本?同じPC?）
□ 緊急時の対応を話し合う
□ 各自のNG曲を共有（被り防止）
□ リハーサルを最低1回実施

合図の例:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
指1本上げ  →「次の曲で交代」
手をパタパタ →「もう少し続けて」
頷き       →「いいね、このまま」
首振り     →「違う方向に行こう」
手で下げる →「エネルギー下げて」
手で上げる →「エネルギー上げて」
```

**B2Bでの曲選びの戦略:**
```
自分のターンで考えること:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 相手が置いた曲のBPM/Keyを即座に確認
   → Rekordboxの表示を見る
   → 耳でBPMを感じ取る

2. 3つの選択肢を常に用意
   ├─ 選択A: 相手の流れを継続（同系統の曲）
   ├─ 選択B: 少し方向転換（サブジャンル変更）
   └─ 選択C: 大胆な展開（ジャンル変更）

3. フロアの反応を基準に選択
   ├─ 盛り上がっている → Aで継続
   ├─ 普通 → Bで新しい要素を
   └─ 微妙 → Cで空気を変える

4. 相手の「得意技」を引き出す
   → 相手がアシッドテクノが得意なら
   → アシッドっぽい曲を振って準備させる
```

### 14.3 B2Bのよくある失敗と対策

```
失敗1: 曲被り
━━━━━━━━━━━━━━━━━━━━━━━━
問題: 同じ曲を両方が持っていて被る
対策: 事前にプレイリスト共有
     → 被った曲はどちらがかけるか決めておく

失敗2: エゴの衝突
━━━━━━━━━━━━━━━━━━━━━━━━
問題: 両方が自分のスタイルを主張しすぎる
対策: 「セット全体」の完成度を最優先にする
     → 個人のプレイより全体の流れ

失敗3: BPMの急変
━━━━━━━━━━━━━━━━━━━━━━━━
問題: 交代時にBPMが急に変わる
対策: ±3 BPM以内のルールを設定
     → 大きく変えたい場合はブレイクダウン利用

失敗4: 機材トラブル
━━━━━━━━━━━━━━━━━━━━━━━━
問題: USB認識しない、ソフトがクラッシュ
対策: バックアップUSB持参
     → 相手が繋いでいる間にトラブル対応
     → 最悪の場合、相手に長めにプレイしてもらう

失敗5: コミュニケーション不足
━━━━━━━━━━━━━━━━━━━━━━━━
問題: 相手の意図が読めず、チグハグな展開に
対策: 大げさなくらいのジェスチャー
     → 暗い中でも見える合図を使う
     → 曲を選ぶ時に相手に画面を見せる
```

## 15. フェスティバルセットの特殊性

### 15.1 フェスティバルセットと通常クラブセットの違い

**環境の違い:**
```
┌──────────────────────────────────────────────────────┐
│ 要素         クラブ             フェスティバル         │
├──────────────────────────────────────────────────────┤
│ 音響環境     最適化されている   屋外、風の影響あり     │
│ 視覚演出     照明中心           映像・花火・CO2等     │
│ 観客規模     50-2,000人        1,000-100,000人       │
│ 観客の状態   アルコール中心     多様（昼間は素面も）   │
│ 持ち時間     60-180分          30-90分が多い          │
│ 前後のDJ     1-2人             多数のアクトが同日     │
│ 音量         一定               PA依存、変動あり      │
│ 時間帯       深夜中心           昼〜深夜まで多様      │
│ プレッシャー  中                高（大観衆の前）       │
│ 機材確認     事前に可能         直前まで不可能な場合も │
└──────────────────────────────────────────────────────┘
```

### 15.2 フェスティバルセットの設計原則

**時間帯別アプローチ:**
```
昼間セット（12:00-16:00）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- 明るい、ポジティブなバイブス
- BPM: やや控えめ（120-126）
- キャッチーなボーカルトラック
- 観客はリラックスモード
- 「日光の下で映える曲」を選ぶ
- ハードすぎない、爽やかなサウンド

夕方セット（16:00-20:00）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- サンセットの雰囲気を活かす
- BPM: 中程度（124-130）
- メロディックで感動的な曲
- エモーショナルなビルドアップ
- 「夕焼けに映える曲」がキー
- プログレッシブ/メロディック系が最適

夜セット（20:00-24:00）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- フルパワー、最高のエネルギー
- BPM: 高め（126-135+）
- ドロップ重視、インパクト大の曲
- 照明・映像との連携を意識
- アンセムの投入タイミングが重要
- フロアのキャパシティが最大

深夜セット（24:00-）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- コアなファンが残る時間帯
- 実験的な選曲が許される
- ディープ/アンダーグラウンドに寄せる
- 長いミックス、催眠的なグルーヴ
- 「知る人ぞ知る」曲で差別化
```

### 15.3 フェスティバル特有のテクニック

**大きな会場でのBPM/キー管理:**
```
PA遅延の問題:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
大きなフェスティバルではPAスピーカーの距離により
音の遅延が発生する。

影響:
- 前方と後方で聴こえるビートにズレが生じる
- ミックスのタイミングがモニターと客席で異なる

対策:
1. モニタースピーカーの音量を十分に確保
2. ヘッドフォンモニタリングを重視
3. ビートマッチはモニターを基準に
4. ロングミックスより短めのトランジションが安全
```

**フェスティバル限定の演出テクニック:**
```
1. 「ハンズアップ」モーメント
   → 全員が手を上げる瞬間を意図的に作る
   → ブレイクダウン → ビルドアップ → ドロップの黄金パターン
   → 1セットに3-5回程度が適切

2. マッシュアップの活用
   → 2つの有名曲を重ねることで歓声を誘う
   → 例: アカペラ + インストの組み合わせ
   → 事前に準備したエディットを使用

3. MCコール
   → 「Make some noise!」的なサンプル
   → 使いすぎ注意（1セット2-3回まで）
   → フロアとの一体感を作る

4. サプライズ要素
   → 未発表曲のプレミア
   → ゲストDJの飛び入り
   → 予想外のジャンル投入
```

## 16. 予備曲の戦略的選択

### 16.1 予備曲プールの設計

予備曲は「保険」ではなく、セットの質を高めるための「戦略的武器」です。

**予備曲の分類体系:**
```
カテゴリA: エネルギーブースター（緊急盛り上げ用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
数量: 5-8曲
特徴: 誰もが反応する定番アンセム
用途: フロアが冷えた時の起爆剤
BPM: セットのターゲットBPM範囲内

選曲基準:
- イントロが短い（すぐに効果が出る）
- ドロップが印象的
- ボーカルフックが強い
- 誰もが知っている（認知度が高い）

カテゴリB: ムードチェンジャー（方向転換用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
数量: 5-8曲
特徴: ジャンル/ムードを変えられる曲
用途: 現在の方向が合っていない時の修正
BPM: 幅広い範囲をカバー

選曲基準:
- ジャンルの境界にある曲
- BPMが柔軟に調整可能
- トランジションしやすい構造
- ムードの切り替えが明確

カテゴリC: セーフティネット（安全策）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
数量: 5曲
特徴: 絶対に外さない鉄板曲
用途: パニック時の最後の砦
BPM: セットのメインBPM

選曲基準:
- 過去に100%の成功率を誇る曲
- どんなフロアでも機能する普遍性
- 自分が最も自信を持ってプレイできる曲
- ミックスイン/アウトが容易

カテゴリD: エクスペリメンタル（実験用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
数量: 3-5曲
特徴: 新曲、レア曲、自作曲
用途: フロアの反応が良い時に挑戦
BPM: 様々

選曲基準:
- 最近入手した新しい曲
- 自分のオリジナルトラック/リミックス
- あまり知られていないが質の高い曲
- リスクはあるが当たれば大きい曲
```

### 16.2 BPM/Key別の予備曲管理

**予備曲マトリックス:**
```
BPM範囲ごとに各カテゴリの予備を用意:

          カテゴリA  カテゴリB  カテゴリC  カテゴリD
BPM 118-122  2曲       2曲       1曲       1曲
BPM 123-126  2曲       2曲       2曲       1曲
BPM 127-130  3曲       2曲       2曲       2曲
BPM 131-135  2曲       2曲       1曲       1曲

Key別の整理:
- 使用頻度の高いキー（8A,9A,10A）: 各キー3曲以上
- 使用頻度の低いキー: 各キー1-2曲

管理方法:
Rekordbox/Seratoのプレイリスト内に
「_BACKUP_A」「_BACKUP_B」等のフォルダを作成
→ 色分けタグで視覚的に区別
```

### 16.3 予備曲投入の判断基準

**投入判断のフローチャート:**
```
現状を評価
    │
    ├─ フロアの反応が期待以下
    │   │
    │   ├─ 「冷えている」→ カテゴリA（エネルギーブースター）
    │   ├─ 「方向が違う」→ カテゴリB（ムードチェンジャー）
    │   └─ 「パニック」  → カテゴリC（セーフティネット）
    │
    ├─ フロアの反応が良好
    │   │
    │   ├─ 「最高潮」    → セットリスト続行
    │   └─ 「余裕あり」  → カテゴリD（実験曲を試す）
    │
    └─ 時間が余る/足りない
        │
        ├─ 余る → 予備曲から追加
        └─ 足りない → 曲をスキップ、ミックス短縮
```
