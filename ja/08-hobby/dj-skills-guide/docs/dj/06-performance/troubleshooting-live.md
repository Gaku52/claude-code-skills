## なぜトラブル対応が重要か

**パニック vs 冷静:**
```
パニック:
- トラブル発生
- 焦る
- さらに悪化
- 観客が不安
- 失敗

冷静な対応:
- トラブル発生
- 深呼吸
- 手順通り対処
- 観客は気づかない
- 成功

トラブル対応 = プロの証
```

## 2. 機材トラブル対応

### トラブル1: USB切断（最も多い）

**症状:**
```
プレイ中に:
- 音が突然止まる
- DDJ-FLX4のLED消える
- Rekordboxが「デバイス未接続」表示

原因:
- USBケーブルが抜けた
- ケーブル不良
- 踊って振動
```

**対処法（30秒以内）:**
```
1. 深呼吸（2秒）
   - 落ち着く
   - 焦らない

2. USBケーブル確認（5秒）
   - 抜けていないか
   - しっかり差し込む

3. 再接続（10秒）
   - ケーブル抜く
   - 再度差し込む
   - DDJ-FLX4のLED点灯確認

4. Rekordbox確認（5秒）
   - 「DDJ-FLX4」認識されているか
   - されていればOK

5. プレイ再開（5秒）
   - 曲をロード
   - 再生開始

6. 笑顔（3秒）
   - 観客に向かって
   - 「大丈夫」アピール

合計: 30秒

練習しておく
```

**予防:**
```
✓ USBケーブルをテープで固定
✓ 予備USBケーブル準備
✓ ケーブルを定期的に交換
✓ 激しく動かない
```

### トラブル2: ヘッドフォン故障

**症状:**
```
- 片方から音が出ない
- 全く聞こえない
- ノイズが入る

原因:
- ケーブル断線
- プラグ不良
- ヘッドフォン故障
```

**対処法:**
```
1. プラグ確認
   - しっかり差さっているか
   - 抜き差し

2. 別のヘッドフォン端子試す
   - DDJ-FLX4には1つだけ
   - 音響スタッフに予備借りる

3. モニタースピーカー使用
   - ヘッドフォンなしでミックス
   - 上級スキル必要

4. 予備ヘッドフォン
   - 持参していれば交換

予備ヘッドフォン必須
```

### トラブル3: DDJ-FLX4フリーズ

**症状:**
```
- ボタンが反応しない
- LED点滅
- 音が出ない

原因:
- ファームウェアバグ
- PCの負荷
- 電源不良
```

**対処法:**
```
1. USB抜き差し
   - 再接続
   - 5秒待つ

2. DDJ-FLX4再起動
   - 電源OFF
   - 5秒待つ
   - 電源ON

3. PC再起動（最終手段）
   - Rekordbox終了
   - PC再起動
   - 2-3分かかる

4. 予備PCに切り替え
   - 持参していれば

再起動 = 2-3分の無音
観客に説明必要
```

## 3. ソフトウェアトラブル対応

### トラブル4: Rekordboxクラッシュ

**症状:**
```
- Rekordbox突然終了
- 「応答なし」表示
- 音が止まる

原因:
- メモリ不足
- ソフトウェアバグ
- PCスペック不足
```

**対処法:**
```
1. Rekordbox強制終了（macOS）
   - Command + Option + Esc
   - Rekordbox選択
   - 「強制終了」

1. Rekordbox強制終了（Windows）
   - Ctrl + Alt + Delete
   - タスクマネージャー
   - Rekordbox選択
   - 「タスクの終了」

2. Rekordbox再起動
   - アプリを再度起動
   - 30秒かかる

3. DDJ-FLX4認識確認
   - 左上に表示

4. プレイ再開
   - セットリストから再開

合計: 1-2分

この間、音響スタッフに曲をかけてもらう
```

**予防:**
```
✓ PCのメモリ8GB以上
✓ 他のアプリを全て閉じる
✓ Rekordboxを最新版に
✓ PCを定期的に再起動
```

### トラブル5: 音が出ない

**症状:**
```
- 曲は再生中
- レベルメーター動いている
- でもスピーカーから音が出ない

原因:
- Master Level が0
- オーディオ設定ミス
- ケーブル抜け
```

**対処法:**
```
チェックリスト（30秒）:

□ Master Level確認
   - DDJ-FLX4のMaster Levelノブ
   - 上がっているか

□ Channel Fader確認
   - Deck Aのフェーダー上がっているか

□ Trim（ゲイン）確認
   - 0になっていないか

□ RCAケーブル確認
   - Master Out から抜けていないか

□ Rekordboxオーディオ設定
   - Preferences > Audio
   - Audio Device: DDJ-FLX4

□ 音響スタッフに確認
   - ミキサー側の問題か

1つずつ確認
必ず見つかる
```

## 4. 音響トラブル対応

### トラブル6: 音が歪む

**症状:**
```
- 音がバリバリ
- クリッピング
- 聞くに耐えない

原因:
- ゲイン大きすぎ
- Master Level大きすぎ
- EQブースト過多
```

**対処法:**
```
1. レベルメーター確認
   - 赤点灯？
   - 黄色？

2. Trim（ゲイン）を下げる
   - すぐに
   - 緑〜黄色に

3. Master Levelを下げる
   - ゲイン下げても歪むなら

4. EQ確認
   - ブーストしすぎていないか
   - 12時（中央）に戻す

5. 音響スタッフに相談
   - 「音が歪んでいます」
   - ミキサー側の問題か

即座に対処
```

---

**症状:**
```
- ヘッドフォンから音が聞こえない
- または小さすぎ

原因:
- Headphone Levelが0
- Mixingノブが間違った位置
- ヘッドフォン故障
```

**対処法:**
```
1. Headphone Level確認
   - DDJ-FLX4の音量ノブ
   - 上げる

2. Cueボタン確認
   - 点灯しているか
   - 押す

3. Mixingノブ確認
   - Cue側に回す

4. ヘッドフォン確認
   - プラグしっかり差さっているか

5. 別のヘッドフォン試す
   - 予備使用

モニターなし = ミックス不可能
最優先で解決
```

## 6. 冷静さを保つ方法

### トラブル時のマインドセット

**心構え:**
```
事実:
- トラブルは必ず起こる
- プロDJも経験
- 誰も完璧じゃない

重要:
✓ トラブルそのものより対応
✓ パニックが最悪
✓ 冷静さ = プロ

観客の視点:
- 詳細はわからない
- DJが焦っているかどうかだけ
- 笑顔なら問題ないと思う

冷静 = プロ
```

### 4ステップ対処法

**STOP法:**
```
S: Stop（止まる）
   - 深呼吸
   - 2秒

T: Think（考える）
   - 何が問題か
   - 3秒

O: Options（選択肢）
   - どう対処するか
   - 3秒

P: Proceed（実行）
   - 落ち着いて実行
   - 20秒

合計: 30秒

練習しておく
```

## 12. ライブ中のトラブル対処フローチャート

### 緊急対応フローチャート

**音が止まった場合の5秒診断:**
```
音が止まった
    ↓
レベルメーター動いている？
    ↓
YES → 音響システムの問題
    ├─ Master Level確認
    ├─ ケーブル確認
    └─ 音響スタッフに連絡

NO → コントローラー/PCの問題
    ↓
DDJ-FLX4のLED点灯している？
    ↓
YES → Rekordboxの問題
    ├─ オーディオ設定確認
    ├─ Rekordbox再起動
    └─ 予備PC使用

NO → USB接続の問題
    ├─ USBケーブル再接続
    ├─ DDJ-FLX4再起動
    └─ 予備ケーブル使用

診断時間: 5秒
対処時間: 30秒〜2分
```

### トラブル重大度マトリックス

**対処優先度:**
```
【緊急度：高】【影響：大】
→ 即座に対処（10秒以内）
- 音が完全に止まった
- スピーカーから煙が出た
- フィードバックハウリング

【緊急度：高】【影響：中】
→ すぐに対処（30秒以内）
- USB切断
- 音が歪む
- モニターが聞こえない

【緊急度：中】【影響：中】
→ 曲終わりに対処（2分以内）
- ヘッドフォン片方故障
- ソフトウェアの小さなバグ
- レイテンシー増加

【緊急度：低】【影響：小】
→ セット後に対処
- LED一部不点灯
- 表示の乱れ
- マイナーな不具合
```

## 13. 音声トラブル詳細対応

### ノイズトラブル

**症状と原因:**
```
ノイズの種類:

1. ハムノイズ（ブーン音）
   原因: グラウンドループ
   周波数: 50/60Hz

2. ホワイトノイズ（サー音）
   原因: ゲイン高すぎ、ケーブル不良

3. デジタルノイズ（プチプチ音）
   原因: バッファサイズ小、CPU負荷

4. 干渉ノイズ（ピーピー音）
   原因: 携帯電話、Wi-Fi干渉
```

**即時対応（ノイズ別）:**
```
ハムノイズ対処:
1. グラウンドリフトアダプター使用
2. 別の電源コンセント試す
3. ケーブルを離す（電源と音声）
4. 音響スタッフに連絡

ホワイトノイズ対処:
1. ゲインを下げる
2. ケーブル接続確認
3. 別のケーブルで試す
4. シールドされたケーブル使用

デジタルノイズ対処:
1. バッファサイズ増やす
   Rekordbox > Preferences > Audio
   512 samples以上
2. 他のアプリ終了
3. CPUパフォーマンスモード
4. USB3.0ポート使用

干渉ノイズ対処:
1. スマホを機内モード
2. Wi-Fiオフ
3. ケーブルを機器から離す
4. フェライトコア使用

対処時間: 1-3分
```

### フィードバック対応

**フィードバックハウリング:**
```
症状:
- 「キーン」という高音
- スピーカーから大音量
- 耳が痛い

原因:
- マイクとスピーカーの位置
- モニターレベル高すぎ
- EQブースト過多

緊急対処（5秒以内）:
1. Master Levelすぐ下げる
   - 即座に
   - フェーダーを0に

2. 問題特定（10秒）
   - マイク使用中？
   - モニターレベル？
   - EQブースト？

3. 恒久対策
   - マイクの位置変更
   - モニターレベル下げる
   - EQ調整（問題周波数カット）
   - ノッチフィルター使用

4. ゆっくりレベル戻す
   - 少しずつ
   - フィードバック再発確認

フィードバック = 観客に大迷惑
最優先で止める
```

### 音が左右どちらかから出ない

**ステレオトラブル:**
```
症状:
- 片方のスピーカーから音が出ない
- モノラルで聞こえる

原因チェック:
□ RCAケーブル片方抜け
□ ケーブル断線
□ チャンネル設定ミス
□ スピーカー故障

対処手順:
```

```python
from typing import List, Dict, Optional
from datetime import datetime

class AdvancedExercise:
    """応用パターンの演習"""

    def __init__(self, max_size: int = 100):
        self._items: List[Dict] = []
        self._max_size = max_size
        self._created_at = datetime.now()

    def add(self, key: str, value: any) -> bool:
        """アイテムの追加（サイズ制限付き）"""
        if len(self._items) >= self._max_size:
            return False
        self._items.append({
            'key': key,
            'value': value,
            'timestamp': datetime.now().isoformat()
        })
        return True

    def find(self, key: str) -> Optional[Dict]:
        """キーによる検索"""
        for item in reversed(self._items):
            if item['key'] == key:
                return item
        return None

    def remove(self, key: str) -> bool:
        """キーによる削除"""
        for i, item in enumerate(self._items):
            if item['key'] == key:
                self._items.pop(i)
                return True
        return False

    def stats(self) -> Dict:
        """統計情報"""
        return {
            'total_items': len(self._items),
            'max_size': self._max_size,
            'usage_percent': len(self._items) / self._max_size * 100,
            'uptime': str(datetime.now() - self._created_at)
        }

# テスト
```

```python
def slow_search(data: list, target: int) -> int:
    """非効率な検索"""
    for i in range(len(data)):
        for j in range(i + 1, len(data)):
            if data[i] + data[j] == target:
                return (i, j)
    return (-1, -1)

# 最適化後（O(n)）
def fast_search(data: list, target: int) -> tuple:
    """ハッシュマップを使った効率的な検索"""
    seen = {}
    for i, num in enumerate(data):
        complement = target - num
        if complement in seen:
            return (seen[complement], i)
        seen[num] = i
    return (-1, -1)
```

```python
    """関数の入出力をログ出力するデコレータ"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        logger.debug(f"呼び出し: {func.__name__}(args={args}, kwargs={kwargs})")
        try:
            result = func(*args, **kwargs)
            logger.debug(f"戻り値: {func.__name__} -> {result}")
            return result
        except Exception as e:
            logger.error(f"例外発生: {func.__name__}: {e}")
            logger.error(traceback.format_exc())
            raise
    return wrapper

@debug_decorator
```

### トレードオフの分析

技術的な判断には必ずトレードオフが伴います。以下の観点で分析を行いましょう:

**1. 短期 vs 長期のコスト**
- 短期的に速い方法が長期的には技術的負債になることがある
- 逆に、過剰な設計は短期的なコストが高く、プロジェクトの遅延を招く

**2. 一貫性 vs 柔軟性**
- 統一された技術スタックは学習コストが低い
- 多様な技術の採用は適材適所が可能だが、運用コストが増加

**3. 抽象化のレベル**
- 高い抽象化は再利用性が高いが、デバッグが困難になる場合がある
- 低い抽象化は直感的だが、コードの重複が発生しやすい

```python
# 設計判断の記録テンプレート
class ArchitectureDecisionRecord:
    """ADR (Architecture Decision Record) の作成"""

    def __init__(self, title: str):
        self.title = title
        self.context = ""
        self.decision = ""
        self.consequences = []
        self.alternatives = []

    def set_context(self, context: str):
        """背景と課題の記述"""
        self.context = context
        return self

    def set_decision(self, decision: str):
        """決定内容の記述"""
        self.decision = decision
        return self

    def add_consequence(self, consequence: str, positive: bool = True):
        """結果の追加"""
        self.consequences.append({
            'description': consequence,
            'type': 'positive' if positive else 'negative'
        })
        return self

    def add_alternative(self, name: str, reason_rejected: str):
        """却下した代替案の追加"""
        self.alternatives.append({
            'name': name,
            'reason_rejected': reason_rejected
        })
        return self

    def to_markdown(self) -> str:
        """Markdown形式で出力"""
        md = f"# ADR: {self.title}\n\n"
        md += f"## 背景\n{self.context}\n\n"
        md += f"## 決定\n{self.decision}\n\n"
        md += "## 結果\n"
        for c in self.consequences:
            icon = "✅" if c['type'] == 'positive' else "⚠️"
            md += f"- {icon} {c['description']}\n"
        md += "\n## 却下した代替案\n"
        for a in self.alternatives:
            md += f"- **{a['name']}**: {a['reason_rejected']}\n"
        return md
```

### 一般的な脆弱性と対策

| 脆弱性 | リスクレベル | 対策 | 検出方法 |
|--------|------------|------|---------|
| インジェクション攻撃 | 高 | 入力値のバリデーション・パラメータ化クエリ | SAST/DAST |

| 機密データの露出 | 高 | 暗号化・アクセス制御 | セキュリティ監査 |

| ログの不足 | 中 | 構造化ログ・監査証跡 | ログ分析 |

### セキュアコーディングのベストプラクティス

```python
# セキュアコーディング例
import hashlib
import secrets
import hmac
from typing import Optional

class SecurityUtils:
    """セキュリティユーティリティ"""

    @staticmethod
    def generate_token(length: int = 32) -> str:
        """暗号学的に安全なトークン生成"""
        return secrets.token_urlsafe(length)

    @staticmethod
    def hash_password(password: str, salt: Optional[str] = None) -> tuple:
        """パスワードのハッシュ化"""
        if salt is None:
            salt = secrets.token_hex(16)
        hashed = hashlib.pbkdf2_hmac(
            'sha256',
            password.encode('utf-8'),
            salt.encode('utf-8'),
            iterations=100000
        )
        return hashed.hex(), salt

    @staticmethod
    def verify_password(password: str, hashed: str, salt: str) -> bool:
        """パスワードの検証"""
        new_hash, _ = SecurityUtils.hash_password(password, salt)
        return hmac.compare_digest(new_hash, hashed)

    @staticmethod
    def sanitize_input(value: str) -> str:
        """入力値のサニタイズ"""
        dangerous_chars = ['<', '>', '"', "'", '&', '\\']
        result = value
        for char in dangerous_chars:
            result = result.replace(char, '')
        return result.strip()

# 使用例
token = SecurityUtils.generate_token()
hashed, salt = SecurityUtils.hash_password("my_password")
is_valid = SecurityUtils.verify_password("my_password", hashed, salt)
```

### 段階的移行の手順

```python
# マイグレーションスクリプトのテンプレート
import json
import logging
from pathlib import Path
from datetime import datetime
from typing import List, Dict, Callable

logger = logging.getLogger(__name__)

class MigrationRunner:
    """段階的マイグレーション実行エンジン"""

    def __init__(self, migration_dir: str):
        self.migration_dir = Path(migration_dir)
        self.migrations: List[Dict] = []
        self.completed: List[str] = []

    def register(self, version: str, description: str,
                 up: Callable, down: Callable):
        """マイグレーションの登録"""
        self.migrations.append({
            'version': version,
            'description': description,
            'up': up,
            'down': down,
            'registered_at': datetime.now().isoformat()
        })

    def run_up(self, target_version: str = None):
        """マイグレーションの実行（アップグレード）"""
        for migration in self.migrations:
            if migration['version'] in self.completed:
                continue
            logger.info(f"実行中: {migration['version']} - "
                       f"{migration['description']}")
            try:
                migration['up']()
                self.completed.append(migration['version'])
                logger.info(f"完了: {migration['version']}")
            except Exception as e:
                logger.error(f"失敗: {migration['version']}: {e}")
                raise
            if target_version and migration['version'] == target_version:
                break

    def run_down(self, target_version: str):
        """マイグレーションのロールバック"""
        for migration in reversed(self.migrations):
            if migration['version'] not in self.completed:
                continue
            if migration['version'] == target_version:
                break
            logger.info(f"ロールバック: {migration['version']}")
            migration['down']()
            self.completed.remove(migration['version'])

    def status(self) -> Dict:
        """マイグレーション状態の確認"""
        return {
            'total': len(self.migrations),
            'completed': len(self.completed),
            'pending': len(self.migrations) - len(self.completed),
            'versions': {
                m['version']: 'completed'
                if m['version'] in self.completed else 'pending'
                for m in self.migrations
            }
        }
```
