## 3. DDJ-FLX4セットアップ手順

### Step 1: 機材の配置

**ブースでの配置:**
```
1. DJブースに入る
   - 前のDJが終わるのを待つ
   - スペース確認

2. DDJ-FLX4を設置
   場所:
   - ブースの中央
   - 安定した場所
   - 手が届きやすい高さ

3. ノートPCを配置
   - DDJ-FLX4の後ろ
   - 画面が見える位置
   - 安定した場所

注意:
✗ 飲み物をコントローラーの近くに置かない
✗ 不安定な場所
✗ ケーブルが邪魔な位置
```

### Step 2: ケーブル接続

**接続順序（重要）:**
```
正しい順序:

1. ノートPC電源接続（バッテリー保護）
2. DDJ-FLX4とPCをUSBケーブル接続
   - DDJ-FLX4背面のUSB端子
   - PCのUSB端子
   - しっかり差し込む

3. ヘッドフォン接続
   - DDJ-FLX4前面のヘッドフォン端子
   - 6.3mm標準ジャックまたは3.5mm

4. DDJ-FLX4の電源を入れる
   - 背面の電源スイッチON
   - LEDが点灯

5. クラブのミキサーまたはスピーカーに接続
   （DDJ-FLX4の場合、クラブのシステムに接続する場合）
   - Master Out（背面）からミキサーへ
   - RCAケーブル（赤/白）

USB給電の場合:
- AC電源不要
- PCから給電
- ケーブル1本で完結
```

### Step 3: Rekordbox起動と設定

**Rekordbox 7.2.7:**
```
1. Rekordboxを起動
   - アプリケーション起動
   - 数秒待つ

2. DDJ-FLX4が認識されているか確認
   - 左上に「DDJ-FLX4」表示
   - もし表示されない:
     → Preferences > Audio
     → Audio Device: DDJ-FLX4選択

3. オーディオ設定確認
   Preferences > Audio:

   ✓ Audio Device: DDJ-FLX4
   ✓ Sample Rate: 44100 Hz（推奨）
   ✓ Buffer Size: 256 samples（推奨）

   Buffer Size:
   - 小さい（128）: レスポンス速い、CPU負担大
   - 大きい（512）: レスポンス遅い、CPU負担小
   - 256が最適

4. キーボードショートカット無効化（重要）
   Preferences > Controller:
   ✓ Keyboard Shortcuts: OFF

   理由: 誤操作防止
```

## 4. サウンドチェック

### サウンドチェックとは

```
サウンドチェック = 音量レベルの最適化

目的:
- Master音量の確認
- 各チャンネルのゲイン調整
- モニターバランス確認
- 音質チェック

時間: 5-10分
絶対に省略しない
```

### Step 1: Master音量確認

**基本手順:**
```
1. テスト曲を1曲ロード
   - Deck Aに
   - よく知っている曲
   - 音圧が標準的な曲

2. Deck Aを再生
   - Playボタン押す
   - 音が出ない（まだMasterが0）

3. Channel Fader（縦フェーダー）を上げる
   - Deck Aのフェーダーを上（MAX）

4. Trimノブ（ゲイン）を調整
   - まず12時（中央）
   - レベルメーター確認

5. レベルメーター確認
   DDJ-FLX4のレベルメーター:
   - 緑: 良い（-18dB ~ -6dB）
   - 黄色: 注意（-6dB ~ -3dB）
   - 赤: 大きすぎ（0dB以上）

   目標: 緑〜黄色の境界（-6dB付近）

6. Master Level（Master音量）を上げる
   - 徐々に上げる
   - クラブスタッフに確認
   「この音量で大丈夫ですか？」

7. 適切な音量に調整
   - 大きすぎず
   - 小さすぎず
   - スタッフのOKをもらう
```

### Step 2: 両デッキの音量を揃える

**ゲインマッチング:**
```
1. Deck Aでテスト曲再生中

2. Deck Bに別の曲をロード

3. Deck Bのヘッドフォンキューボタン押す
   - ヘッドフォンでDeck Bを聴く
   - まだスピーカーからは出ない

4. Deck BのTrim（ゲイン）を調整
   - レベルメーターを見る
   - Deck Aと同じレベルに
   - 緑〜黄色の境界

5. 両デッキを比較
   - Deck A再生
   - Deck Bに切り替え
   - 音量が同じか確認

6. 全ての曲で繰り返す
   （本番中も常にゲイン調整）

ゲインが揃っている = プロ
```

## 5. モニターバランス調整

### モニターとは

```
モニター = ヘッドフォンから聞こえる音

2つの音源:
1. Master（スピーカーから出ている音）
2. Cue（次にプレイする曲）

ミックス比を調整可能
```

### DDJ-FLX4でのモニターバランス

**Mixing（ミキシング）ノブ:**
```
DDJ-FLX4のヘッドフォンセクション:

Mixingノブ（中央上部）:
- 左（Master）: スピーカーの音のみ
- 中央: Master + Cue 50/50
- 右（Cue）: 次の曲のみ

おすすめ設定:

初心者:
→ 右（Cue）寄り
理由: 次の曲をしっかり聞く

中級者:
→ 中央
理由: 両方バランスよく

上級者:
→ 状況に応じて調整

自分に合った設定を見つける
```

### ヘッドフォン音量調整

**Headphone Levelノブ:**
```
DDJ-FLX4のヘッドフォン音量:

Level（音量）ノブ:
- 左: 小さい
- 右: 大きい

適切な音量:
✓ 曲のビートがはっきり聞こえる
✓ 会場の音も少し聞こえる
✓ 耳が痛くならない

大きすぎ:
✗ 耳を痛める
✗ 会場の音が聞こえない
✗ 観客の反応がわからない

小さすぎ:
✗ ビートマッチできない
✗ 音質確認できない

中間〜やや大きめが最適
```

## 6. クラブでの標準的なセットアップ

### CDJセットアップとの違い

**DDJ-FLX4の利点:**
```
CDJセットアップ（標準的なクラブ）:
- CDJ x 2台
- DJミキサー x 1台
- USBメモリ必須
- 複雑な接続

DDJ-FLX4セットアップ:
- DDJ-FLX4のみ
- ノートPC
- USBケーブル1本
- シンプル

セットアップ時間:
CDJ: 15-20分
DDJ-FLX4: 5-10分

DDJ-FLX4の方が簡単
```

### クラブのサウンドシステムへの接続

**2つの方法:**
```
方法1: DDJ-FLX4から直接出力（推奨）

接続:
DDJ-FLX4背面 Master Out（RCA）
→ クラブのミキサーまたはアンプ

利点:
✓ シンプル
✓ トラブル少ない
✓ 初心者向け

方法2: クラブのミキサー経由

接続:
DDJ-FLX4 → クラブのミキサー → スピーカー

利点:
✓ クラブのEQ使える
✓ 他DJと共有しやすい

初心者は方法1推奨
```

## 7. トラブル回避

### よくあるトラブルと対処

**トラブル1: 音が出ない**
```
原因:
- ケーブル接続ミス
- Master Level が0
- Channel Fader が0
- オーディオ設定ミス

対処:
1. 全てのケーブル確認
2. Master Level を上げる
3. Channel Fader を上げる
4. Rekordbox Audio設定確認
5. DDJ-FLX4再起動
```

**トラブル2: 音が歪む**
```
原因:
- Trim（ゲイン）が大きすぎ
- Master Level が大きすぎ
- クリッピング

対処:
1. レベルメーター確認（赤点灯？）
2. Trimを下げる
3. Master Level を下げる
4. EQを使いすぎていないか確認
```

**トラブル3: モニターが聞こえない**
```
原因:
- ヘッドフォン接続ミス
- Headphone Level が0
- Cueボタン押していない

対処:
1. ヘッドフォンしっかり差し込む
2. Headphone Level を上げる
3. Cueボタンが点灯しているか確認
```

**トラブル4: 片方のチャンネルから音が出ない**
```
原因:
- RCAケーブルの片方が抜けている
- ケーブル不良

対処:
1. 両方のRCAケーブル確認（赤/白）
2. 予備ケーブルに交換
3. 別の出力端子を使う
```

## 11. 会場別セットアップガイド

### 小規模バー・カフェ（収容30-50人）

**特徴:**
```
環境:
- 小さなDJスペース
- 壁際またはコーナー配置
- 限られた電源
- PA（音響）システムがシンプル

音響特性:
- 低音が籠もりやすい
- 残響が多い
- 近接音場（お客さんが近い）
```

**セットアップポイント:**
```
機材配置:
✓ コンパクトに収める
✓ 最小限のケーブル
✓ DDJ-FLX4 + ノートPCのみ
✗ 大きなスピーカー不要

接続:
DDJ-FLX4 Master Out
→ バーのミキサーまたは
→ 小型PAシステム（JBL EONなど）

音量設定:
- Master Level: 50-70%程度
- Trim: 控えめ（会話できる音量）
- 低音（Bass）: -2〜-4dB削る
  理由: 小空間では低音が膨らむ

推奨時間配分:
- セットアップ: 10分
- サウンドチェック: 5分
- 計15分
```

### 中規模クラブ（収容100-300人）

**特徴:**
```
環境:
- 専用DJブース
- プロ仕様ミキサー常設
- CDJ設置済み（多くの場合）
- 専用モニタースピーカー

音響特性:
- 低音が強い
- 音響設計されている
- サブウーファー完備
```

**セットアップポイント:**
```
機材配置:
✓ 既存のCDJ/ミキサーと併用可能
✓ DDJ-FLX4を横に配置
✓ 十分なスペース

接続パターン1（DDJ-FLX4のみ使用）:
DDJ-FLX4 Master Out
→ クラブのミキサー Channel 1/2
→ メインPA

接続パターン2（クラブミキサー経由）:
DDJ-FLX4を外部入力として
クラブミキサーのCH経由
→ クラブのEQ/エフェクト使用可能

音量設定:
- Master Level: 70-80%
- Trim: やや強め（クラブ標準）
- レベルメーター: 黄色まで許容
- サブウーファー考慮: 低音調整重要

ブースモニター:
✓ 専用ブースモニター使用
✓ 音量バランス重要
✓ ヘッドフォン + モニター併用

推奨時間配分:
- セットアップ: 15分
- サウンドチェック: 10分
- 計25分
```

### 大規模フェス・野外イベント（収容500人以上）

**特徴:**
```
環境:
- 大型ステージ
- プロ音響チーム常駐
- 複雑なPAシステム
- FOH（Front of House）ミキサー
- 多数のモニター

音響特性:
- 広い空間
- 音の遅延
- 野外の場合は音が拡散
- 強力な低音
```

**セットアップポイント:**
```
機材配置:
✓ ステージマネージャーと連携
✓ 専用DJブース/ステージエリア
✓ 複数のモニター配置
✓ バックアップシステム必須

接続:
DDJ-FLX4 Master Out
→ ステージボックス（XLR変換）
→ FOHミキサー
→ メインPAシステム

重要: RCA to XLR変換器必須
（大型イベントはXLR接続が標準）

音量設定:
- Master Level: FOHエンジニアに一任
- Trim: 標準レベル維持
- レベルメーター: 緑〜黄色キープ
- 絶対にクリッピングさせない

サウンドチェック:
1. FOHエンジニアと打ち合わせ
2. テスト曲を3-4曲用意
   - 静かなイントロ
   - 通常のトラック
   - ヘビーなベースライン
3. エンジニアが調整
4. ブースモニター確認
5. IEM（イヤーモニター）確認（使用する場合）

バックアップ体制:
✓ 予備DDJ-FLX4またはUSBメモリ
✓ 予備ノートPC
✓ 全曲USBメモリにも保存
✓ クラウドバックアップ（Wi-Fi利用可能な場合）

推奨時間配分:
- サウンドチェック専用時間: 30分-1時間
- リハーサル: 15-30分（場合による）
- 本番前最終確認: 10分
```

## 12. DJブースの配線パターンとトラブルシューティング

### 基本配線パターン

**パターン1: ホームセットアップ（最小構成）**
```
ノートPC
  |（USB）
DDJ-FLX4
  |（RCA）
パワードスピーカー（JBL 305P MkII等）

利点:
✓ シンプル
✓ トラブル少ない
✓ 初心者向け
```

**パターン2: クラブセットアップ（標準）**
```
ノートPC
  |（USB）
```

```
  |（Master Out）
```

### 配線トラブルシューティング

**問題1: 音が全く出ない**
```
チェックリスト:
□ 電源が入っているか
  - DDJ-FLX4のLED点灯確認
  - スピーカー/ミキサーの電源確認

□ ケーブル接続確認
  - RCA（赤/白）両方とも差さっているか
  - 入力/出力を間違えていないか
  - ケーブルがしっかり奥まで差さっているか

□ ソフトウェア設定
  - Rekordbox Audio設定確認
  - DDJ-FLX4が選択されているか

□ 音量設定
  - Master Levelが0になっていないか
  - Channel Faderが上がっているか
  - Trimが最小になっていないか

対処:
1. 全てのケーブルを一度抜く
2. 順番に接続し直す
3. 各段階で音を確認
```

**問題2: 片チャンネルのみ音が出ない**
```
原因:
- RCAケーブルの片方（L/R）が抜けている
- ケーブル内部断線
- 接触不良

対処:
1. RCAの赤/白両方を確認
2. ケーブルを交換
3. 別の出力端子を試す
4. スピーカー側の入力も確認
```

**問題3: ノイズ・ハム音が入る**
```
原因:
- グラウンドループ
- ケーブルの品質
- 電源の問題
- 近くの電気機器からの干渉

対処:
1. グラウンドリフトアダプター使用
2. 高品質なケーブルに交換
3. 電源を別系統にする
4. ケーブルを電源コードから離す
5. DDJ-FLX4のグラウンド端子確認
```

**問題4: 音が途切れる・クラックルノイズ**
```
原因:
- Buffer Sizeが小さすぎ
- CPUパワー不足
- USBケーブル不良
- ドライバーの問題

対処:
1. Buffer Size上げる（256 → 512）
2. Rekordbox以外のアプリ終了
3. USBケーブル交換
4. USBポート変更（USB 2.0 → 3.0等）
5. ドライバー再インストール
```

**問題5: レイテンシー（遅延）が大きい**
```
原因:
- Buffer Sizeが大きすぎ
- USB接続の問題

対処:
1. Buffer Size下げる（512 → 256 → 128）
2. 直接USBポートに接続（ハブ経由NG）
3. 高速USBポート使用
4. Sample Rate確認（44100Hz推奨）
```

### ケーブル管理のベストプラクティス

**配線の基本ルール:**
```
1. 電源ケーブルとオーディオケーブルを分ける
   - ノイズ防止
   - 最低30cm離す

2. ケーブルの長さ
   - 必要最小限
   - 長すぎるケーブルは巻かない（ノイズ源）

3. ケーブルの固定
   - ガムテープで床に固定
   - ケーブルタイで束ねる
   - 足で引っ掛けない配置

4. ケーブルの識別
   - 色分けテープで識別
   - ラベリング（L/R、Input/Output）

5. 予備ケーブルの配置
   - すぐ取り出せる場所
   - ケースに整理
```

## 17. サウンドチェック詳細手順とチェックリスト

### プロフェッショナルなサウンドチェックの流れ

**5分間サウンドチェック（標準）:**
```
タイムライン:

0:00-1:00 準備
- テスト曲を3曲選定
  → 低音圧の曲（静かなイントロ）
  → 標準音圧の曲（通常のトラック）
  → 高音圧の曲（ヘビーなビート）
- Deck Aに最初のテスト曲ロード

1:00-2:00 Deck A調整
- 再生開始
- Channel Fader MAX
- Trim調整（緑〜黄色）
- Master Level徐々に上げる
- スタッフに音量確認

2:00-3:00 Deck B調整
- 2曲目をDeck Bにロード
- Cueボタンでモニター
- Trimを調整
- Deck Aと音量を揃える

3:00-4:00 EQ/周波数バランス確認
- 低音（Bass）: 籠もっていないか
- 中音（Mid）: ボーカルが明瞭か
- 高音（High）: シンバルが刺さらないか
- 会場の音響特性を把握

4:00-5:00 最終確認
- 3曲目（高音圧）でクリッピング確認
- モニターバランス最終調整
- ヘッドフォン音量確認
- 「準備完了」をスタッフに報告

効率的かつ完璧
```

### 音響特性別の調整

**小規模会場（バー・カフェ）:**
```
音響特性:
- 低音が籠もる
- 近接音場
- 残響多い
```

## 19. よくあるミスとその対策

### セットアップ時のよくあるミス

**ミス1: ケーブルの接続順序を間違える**
```
間違い:
- DDJ-FLX4の電源を先に入れる
- USBケーブルを後から接続
→ デバイス認識されない

正しい順序:
1. PC電源接続
2. USB接続
3. DDJ-FLX4電源ON
4. Rekordbox起動

ポイント:
USB接続してから電源ON
```

**ミス2: Buffer Sizeを適切に設定しない**
```
問題:
- 小さすぎる（64, 128）
  → 音が途切れる
- 大きすぎる（1024）
  → レイテンシーが大きい

推奨:
256 samples（標準）
512 samples（CPUパワー不足の場合）

確認方法:
Preferences > Audio > Buffer Size
```

**ミス3: ゲイン調整をしない**
```
問題:
- 全曲同じTrim設定
- 音量がバラバラ
→ プロっぽくない

正しい方法:
□ 毎曲ゲイン調整
□ レベルメーター確認
□ 緑〜黄色をキープ

習慣化する
```

**ミス4: モニターバランスを調整しない**
```
問題:
- Mixingノブを初期設定のまま
- ヘッドフォン音量が小さい/大きい
→ ミックスしにくい

正しい設定:
初心者: Cue寄り（次の曲重視）
中級者: 中央（バランス）
上級者: 状況に応じて

自分に合った設定を見つける
```

**ミス5: 予備機材を持ってこない**
```
問題:
- USBケーブルが1本だけ
- USBメモリなし
- 予備ヘッドフォンなし
→ トラブル時に対応できない

必須の予備:
✓ USBケーブル x2
✓ USBメモリ x2（同じ内容）
✓ 予備ヘッドフォン
✓ RCAケーブル（任意）

安心 = パフォーマンス向上
```

```python
class AdvancedExercise:
    """応用パターンの演習"""

    def __init__(self, max_size: int = 100):
        self._items: List[Dict] = []
        self._max_size = max_size
        self._created_at = datetime.now()

    def add(self, key: str, value: any) -> bool:
        """アイテムの追加（サイズ制限付き）"""
        if len(self._items) >= self._max_size:
            return False
        self._items.append({
            'key': key,
            'value': value,
            'timestamp': datetime.now().isoformat()
        })
        return True

    def find(self, key: str) -> Optional[Dict]:
        """キーによる検索"""
        for item in reversed(self._items):
            if item['key'] == key:
                return item
        return None

    def remove(self, key: str) -> bool:
        """キーによる削除"""
        for i, item in enumerate(self._items):
            if item['key'] == key:
                self._items.pop(i)
                return True
        return False

    def stats(self) -> Dict:
        """統計情報"""
        return {
            'total_items': len(self._items),
            'max_size': self._max_size,
            'usage_percent': len(self._items) / self._max_size * 100,
            'uptime': str(datetime.now() - self._created_at)
        }

# テスト
```

## トラブルシューティング

### よくあるエラーと解決策

| エラー | 原因 | 解決策 |
|--------|------|--------|
| 初期化エラー | 設定ファイルの不備 | 設定ファイルのパスと形式を確認 |

| 権限エラー | アクセス権限の不足 | 実行ユーザーの権限確認、設定の見直し |
| データ不整合 | 並行処理の競合 | ロック機構の導入、トランザクション管理 |

### デバッグの手順

1. **エラーメッセージの確認**: スタックトレースを読み、発生箇所を特定する
2. **再現手順の確立**: 最小限のコードでエラーを再現する
3. **仮説の立案**: 考えられる原因をリストアップする
4. **段階的な検証**: ログ出力やデバッガを使って仮説を検証する
5. **修正と回帰テスト**: 修正後、関連する箇所のテストも実行する

```python
# デバッグ用ユーティリティ
import logging
import traceback
from functools import wraps

# ロガーの設定
logging.basicConfig(
    level=logging.DEBUG,
    format='%(asctime)s [%(levelname)s] %(name)s: %(message)s'
)
logger = logging.getLogger(__name__)

def debug_decorator(func):
    """関数の入出力をログ出力するデコレータ"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        logger.debug(f"呼び出し: {func.__name__}(args={args}, kwargs={kwargs})")
        try:
            result = func(*args, **kwargs)
            logger.debug(f"戻り値: {func.__name__} -> {result}")
            return result
        except Exception as e:
            logger.error(f"例外発生: {func.__name__}: {e}")
            logger.error(traceback.format_exc())
            raise
    return wrapper

@debug_decorator
def process_data(items):
    """データ処理（デバッグ対象）"""
    if not items:
        raise ValueError("空のデータ")
    return [item * 2 for item in items]
```

### パフォーマンス問題の診断

パフォーマンス問題が発生した場合の診断手順:

1. **ボトルネックの特定**: プロファイリングツールで計測
2. **メモリ使用量の確認**: メモリリークの有無をチェック
3. **I/O待ちの確認**: ディスクやネットワークI/Oの状況を確認
4. **同時接続数の確認**: コネクションプールの状態を確認

| 問題の種類 | 診断ツール | 対策 |
|-----------|-----------|------|
| CPU負荷 | cProfile, py-spy | アルゴリズム改善、並列化 |
| メモリリーク | tracemalloc, objgraph | 参照の適切な解放 |
| I/Oボトルネック | strace, iostat | 非同期I/O、キャッシュ |
| DB遅延 | EXPLAIN, slow query log | インデックス、クエリ最適化 |

### トレードオフの分析

技術的な判断には必ずトレードオフが伴います。以下の観点で分析を行いましょう:

**1. 短期 vs 長期のコスト**
- 短期的に速い方法が長期的には技術的負債になることがある
- 逆に、過剰な設計は短期的なコストが高く、プロジェクトの遅延を招く

**2. 一貫性 vs 柔軟性**
- 統一された技術スタックは学習コストが低い
- 多様な技術の採用は適材適所が可能だが、運用コストが増加

**3. 抽象化のレベル**
- 高い抽象化は再利用性が高いが、デバッグが困難になる場合がある
- 低い抽象化は直感的だが、コードの重複が発生しやすい

```python
# 設計判断の記録テンプレート
class ArchitectureDecisionRecord:
    """ADR (Architecture Decision Record) の作成"""

    def __init__(self, title: str):
        self.title = title
        self.context = ""
        self.decision = ""
        self.consequences = []
        self.alternatives = []

    def set_context(self, context: str):
        """背景と課題の記述"""
        self.context = context
        return self

    def set_decision(self, decision: str):
        """決定内容の記述"""
        self.decision = decision
        return self

    def add_consequence(self, consequence: str, positive: bool = True):
        """結果の追加"""
        self.consequences.append({
            'description': consequence,
            'type': 'positive' if positive else 'negative'
        })
        return self

    def add_alternative(self, name: str, reason_rejected: str):
        """却下した代替案の追加"""
        self.alternatives.append({
            'name': name,
            'reason_rejected': reason_rejected
        })
        return self

    def to_markdown(self) -> str:
        """Markdown形式で出力"""
        md = f"# ADR: {self.title}\n\n"
        md += f"## 背景\n{self.context}\n\n"
        md += f"## 決定\n{self.decision}\n\n"
        md += "## 結果\n"
        for c in self.consequences:
            icon = "✅" if c['type'] == 'positive' else "⚠️"
            md += f"- {icon} {c['description']}\n"
        md += "\n## 却下した代替案\n"
        for a in self.alternatives:
            md += f"- **{a['name']}**: {a['reason_rejected']}\n"
        return md
```

```python
# セキュアコーディング例
import hashlib
import secrets
import hmac
from typing import Optional

class SecurityUtils:
    """セキュリティユーティリティ"""

    @staticmethod
    def generate_token(length: int = 32) -> str:
        """暗号学的に安全なトークン生成"""
        return secrets.token_urlsafe(length)

    @staticmethod
    def hash_password(password: str, salt: Optional[str] = None) -> tuple:
        """パスワードのハッシュ化"""
        if salt is None:
            salt = secrets.token_hex(16)
        hashed = hashlib.pbkdf2_hmac(
            'sha256',
            password.encode('utf-8'),
            salt.encode('utf-8'),
            iterations=100000
        )
        return hashed.hex(), salt

    @staticmethod
    def verify_password(password: str, hashed: str, salt: str) -> bool:
        """パスワードの検証"""
        new_hash, _ = SecurityUtils.hash_password(password, salt)
        return hmac.compare_digest(new_hash, hashed)

    @staticmethod
    def sanitize_input(value: str) -> str:
        """入力値のサニタイズ"""
        dangerous_chars = ['<', '>', '"', "'", '&', '\\']
        result = value
        for char in dangerous_chars:
            result = result.replace(char, '')
        return result.strip()

# 使用例
token = SecurityUtils.generate_token()
hashed, salt = SecurityUtils.hash_password("my_password")
is_valid = SecurityUtils.verify_password("my_password", hashed, salt)
```

### 段階的移行の手順

```python
# マイグレーションスクリプトのテンプレート
import json
import logging
from pathlib import Path
from datetime import datetime
from typing import List, Dict, Callable

logger = logging.getLogger(__name__)

class MigrationRunner:
    """段階的マイグレーション実行エンジン"""

    def __init__(self, migration_dir: str):
        self.migration_dir = Path(migration_dir)
        self.migrations: List[Dict] = []
        self.completed: List[str] = []

    def register(self, version: str, description: str,
                 up: Callable, down: Callable):
        """マイグレーションの登録"""
        self.migrations.append({
            'version': version,
            'description': description,
            'up': up,
            'down': down,
            'registered_at': datetime.now().isoformat()
        })

    def run_up(self, target_version: str = None):
        """マイグレーションの実行（アップグレード）"""
        for migration in self.migrations:
            if migration['version'] in self.completed:
                continue
            logger.info(f"実行中: {migration['version']} - "
                       f"{migration['description']}")
            try:
                migration['up']()
                self.completed.append(migration['version'])
                logger.info(f"完了: {migration['version']}")
            except Exception as e:
                logger.error(f"失敗: {migration['version']}: {e}")
                raise
            if target_version and migration['version'] == target_version:
                break

    def run_down(self, target_version: str):
        """マイグレーションのロールバック"""
        for migration in reversed(self.migrations):
            if migration['version'] not in self.completed:
                continue
            if migration['version'] == target_version:
                break
            logger.info(f"ロールバック: {migration['version']}")
            migration['down']()
            self.completed.remove(migration['version'])

    def status(self) -> Dict:
        """マイグレーション状態の確認"""
        return {
            'total': len(self.migrations),
            'completed': len(self.completed),
            'pending': len(self.migrations) - len(self.completed),
            'versions': {
                m['version']: 'completed'
                if m['version'] in self.completed else 'pending'
                for m in self.migrations
            }
        }
```

### ロールバック計画
